# FR/EN i18n Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add a runtime FR ↔ EN language toggle for the UI, while making the AI backend language-agnostic so meetings work in French, English, or both.

**Architecture:** A lightweight React context (`LanguageContext`) stores the chosen UI language in `localStorage`. Components call `useT()` hook to get translated strings. The backend prompts replace hardcoded `LANGUE: Réponds TOUJOURS en français` with a dynamic instruction injected at call time based on user preference, plus smart auto-detection for mixed meetings.

**Tech Stack:** React Context API, TypeScript, localStorage, existing Electron IPC pattern

**Step 3: Create `src/i18n/index.ts`**

```typescript
import React, { createContext, useContext, useState, useEffect } from 'react';
import { fr, TranslationKey } from './fr';
import { en } from './en';

type Lang = 'fr' | 'en';

const translations: Record<Lang, Record<TranslationKey, string>> = { fr, en };

interface LanguageContextValue {
  lang: Lang;
  setLang: (l: Lang) => void;
  t: (key: TranslationKey) => string;
}

const LanguageContext = createContext<LanguageContextValue>({
  lang: 'fr',
  setLang: () => {},
  t: (k) => fr[k],
});

export const LanguageProvider: React.FC<{ children: React.ReactNode }> = ({ children }) => {
  const [lang, setLangState] = useState<Lang>(() => {
    return (localStorage.getItem('cluely_ui_lang') as Lang) || 'fr';
  });

  const setLang = (l: Lang) => {
    localStorage.setItem('cluely_ui_lang', l);
    setLangState(l);
  };

  const t = (key: TranslationKey): string => translations[lang][key] ?? fr[key];

  return (
    <LanguageContext.Provider value={{ lang, setLang, t }}>
      {children}
    </LanguageContext.Provider>
  );
};

export const useT = () => useContext(LanguageContext);
```

## Task 5: Make backend prompts language-aware

**Goal:** Replace every `LANGUE: Réponds TOUJOURS en français` with a dynamic instruction that uses the user's stored language preference. Support FR, EN, and auto-detect for mixed meetings.

**Files:**
- Modify: `electron/llm/prompts.ts`
- Modify: `electron/ipcHandlers.ts` (to pass lang to LLM calls)
- Create: `electron/llm/getLanguageInstruction.ts`

**Step 1: Create `electron/llm/getLanguageInstruction.ts`**

```typescript
export type AppLang = 'fr' | 'en' | 'auto';

/**
 * Returns a language instruction to inject into prompts.
 * 'auto' = detect from conversation context and reply in the same language.
 */
export function getLanguageInstruction(lang: AppLang): string {
  switch (lang) {
    case 'fr':
      return 'LANGUE : Réponds TOUJOURS en français, quelle que soit la langue de la question.';
    case 'en':
      return 'LANGUAGE: Always respond in English, regardless of the language of the question.';
    case 'auto':
    default:
      return 'LANGUAGE: Detect the language of the conversation automatically and respond in the same language. If the conversation is mixed (French and English), prefer the language most recently used by the user.';
  }
}
```

## Key decisions

| Decision | Rationale |
|----------|-----------|
| React Context (no library) | Zero deps, sufficient for 2 languages |
| `{{LANG_INSTRUCTION}}` token in prompts | Non-breaking, easy to find/replace, keeps prompt files readable |
| `auto` backend lang mode | Lets meetings in either language work without forcing a choice |
| localStorage + IPC sync | Renderer and main process stay in sync without extra state management |
| French STT already works via `auto` | Google STT detects FR/EN per-utterance — no backend change needed for transcription |
